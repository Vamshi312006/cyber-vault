# Purpose

This document defines the professional roles that this career system is designed to achieve.

Every learning module, flagship project, HTB room, portfolio artifact, certification, and interview preparation activity must contribute toward one or more of these roles.

Roles outside this scope should not receive significant time unless they provide transferable skills.

---

# Primary Target Roles

## Application Security Engineer

### Why

Design, build, review, and secure modern applications.

### Core Responsibilities

- Secure software design
- Authentication & Authorization
- Secure API development
- Threat modeling
- Secure code review
- Vulnerability assessment
- OWASP Top 10 mitigation
- SDLC integration
- Security testing

### Primary Evidence

- Healthcare V2
- Secure coding write-ups
- Threat models
- Security testing

---

## Detection Engineer

### Why

Build systems that detect malicious activity accurately and efficiently.

### Core Responsibilities

- Detection engineering
- Log analysis
- SIEM
- Sigma
- MITRE ATT&CK
- Correlation
- Detection rule development
- Threat hunting
- Incident analysis

### Primary Evidence

- Detection Engineering Platform
- Sigma rules
- Detection writeups
- HTB / SOC labs

# Roles Explicitly Out of Scope

These are not primary career objectives.

- Red Team
- Malware Development
- Exploit Development
- Reverse Engineering
- Mobile Security
- Hardware Security

Learning may occur when useful, but they will not drive roadmap decisions.

---

# Placement Strategy

Priority Order

1. Application Security
2. Detection Engineering
3. Security Engineering
4. Blue Team
5. SOC

Applications should prioritize organizations where flagship projects directly demonstrate required skills.

---

# Decision Rule

Before beginning any significant work, ask:

Does this improve one of the target roles?

If not, reconsider the time investment.
