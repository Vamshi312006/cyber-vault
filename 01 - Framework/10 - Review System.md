# Review System

> Status: Active
> Version: 1.0
> Last Updated:

---

# Purpose

This document defines the review process for the entire career system.

The objective is to ensure continuous improvement through regular evaluation, reflection, and adjustment.

Reviews prevent aimless execution and help maintain alignment with long-term career goals.

---

# Review Philosophy

Execution without review leads to repeated mistakes.

Planning without execution leads to stagnation.

The review system exists to continuously improve both learning and execution.

---

# Review Levels

Daily

↓

Sprint

↓

Monthly

↓

Quarterly

Each level serves a different purpose.

---

# Daily Review

Performed at the end of every work session.

Questions

- What did I complete?
- What did I learn?
- What blocked me?
- What is the next task?

Update:

- CURRENT_STATUS.md

---

# Sprint Review

Performed after completing a sprint.

Evaluate:

- Sprint objective achieved?
- Features completed?
- Skills improved?
- Documentation updated?
- Portfolio evidence created?
- Technical debt introduced?
- Next sprint objective?

---

# Monthly Review

Review overall progress.

Evaluate:

- Career goals
- Skill development
- Project progress
- Cyber Act
- HTB
- Portfolio
- Resume
- Interview readiness

Identify:

- Strengths
- Weaknesses
- Missing skills

---

# Quarterly Review

Review long-term direction.

Questions

- Are the target roles still appropriate?
- Are flagship projects progressing?
- Is the learning strategy effective?
- Should priorities change?

Major strategic changes should only occur during quarterly reviews.

---

# Review Checklist

During every review verify:

✓ Current Status updated

✓ Skill Matrix updated

✓ Project Progress updated

✓ Documentation complete

✓ Resume reviewed

✓ Portfolio reviewed

✓ Next objectives defined

---

# Lessons Learned

Every sprint should produce lessons.

Capture:

- Mistakes
- Better approaches
- Engineering insights
- Security lessons
- Process improvements

Lessons should improve future execution.

---

# Decision Rules

During reviews ask:

- What should continue?
- What should stop?
- What should improve?
- What should begin next?

Avoid making unnecessary architectural changes unless execution exposes a real problem.

---

# Success Criteria

A review is successful when it results in:

- Clear understanding of current progress
- Identification of weaknesses
- Defined next priorities
- Improved execution

---

# Final Principle

Reviews exist to improve future execution.

Their purpose is not to judge past performance but to make the next sprint more effective.
