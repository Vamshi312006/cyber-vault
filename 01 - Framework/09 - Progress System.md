---

# Purpose

This document defines how progress is measured throughout the career system.

Progress should reflect engineering growth rather than time invested.

The objective is to measure capability, not activity.

---

# Measurement Philosophy

Progress is determined by completed capability.

The following are **not** considered reliable indicators of progress:

- Hours studied
- Videos watched
- Courses completed
- Certificates earned
- Number of GitHub repositories

Instead, progress is measured through demonstrated engineering ability.

---

Each skill should be evaluated based on:

- Conceptual understanding
- Practical implementation
- Testing
- Documentation
- Interview confidence
- Portfolio evidence

# Cyber Act Progress

Track:

- Current Stage
- Current Module
- Current Skill
- Completed Reviews

Progress should reflect understanding rather than note completion.

---

# HTB Progress

Track:

- Rooms completed
- Skills reinforced
- ATT&CK techniques observed
- Lessons documented

Room count alone is not meaningful.

---

# Portfolio Progress

- Documentation quality
- GitHub quality

- Technical writing
- Interview preparation

---

# Placement Progress

Track:

- Resume readiness
- Portfolio readiness
- Technical readiness
- Interview readiness
- Applications
- Interviews
- Offers

---

# Progress Review

Progress should be reviewed after every sprint.

Questions:

- What improved?
- What remains incomplete?
- What evidence was created?
- Which skills need reinforcement?
- What is the next priority?

---

# Success Criteria

Progress is meaningful only when it results in increased engineering capability.

Every review should answer one question:

**"Am I more capable than I was during the previous sprint?"**

If the answer is yes, progress has been made.

---

# Final Principle

Progress is measured by capability gained, not effort spent.

Engineering ability—not activity—is the standard by which progress is evaluated.
